## はじめに

iPhoneの写真アプリの検索窓に「ラーメン」と入力すると、過去に撮ったラーメンの写真が並びます。自分でタグ付けをしていなくても、写真アプリが画像の中身を認識して食べ物を探し当てています。

![iPhoneの写真アプリで「ラーメン」と検索した結果](/images/jidoshokki-food-recognition/photos-ramen-search.png)
*写真アプリで「ラーメン」と検索した結果。タグ付けをしていないのに、ラーメンの写真だけが並ぶ。*

この仕組みを応用すれば、すでにアルバムにある写真から食事を自動で拾い、後からまとめて記録できます。撮る前に身構える必要も、手入力の手間もありません。この発想から作ったのが、食事記録アプリ「自動食記（じどうしょっき）」です。

この記事は、追加の学習モデルに頼らず、iOS標準のVision Frameworkだけで、写真から食べ物の種類をどこまで判定できるかを試した記録です。

## 最初の誤算：あの検索機能は開発者に開放されていない

写真アプリで「ラーメン」を検索できるのは、端末の中で写真を解析して、写っているシーンや物体を認識しているからです。Appleはこのオンデバイスのシーン解析について[研究内容を公開](https://machinelearning.apple.com/research/on-device-scene-analysis)していて、写真を一枚ずつ端末内のニューラルネットワークで分類し、その結果を写真の検索などに使っていることがわかります。

ただ、写真アプリの検索を支えるこの高精度なモデルは、そのままの形では開発者に開放されていません。アプリの側から「ラーメンの写真をください」と頼むことはできません。地味に感動したあの体験を、自分のアプリでそのまま再現することはできませんでした。

代わりに使ったのが、iOS標準のVision Framework、具体的には [`VNClassifyImageRequest`](https://developer.apple.com/documentation/vision/vnclassifyimagerequest) です。これは汎用の画像分類器です。`knownClassifications(forRevision:)` で取得できるラベルを数えてみると、revision 2 では1303個ありました。完全にオンデバイスで動き、無料で、API制限もありません。写真をいっさい外部に送らないため、プライバシーの面でも安心して使えます。

条件だけを見ると申し分ないのですが、Appleの非公開モデルと比べると精度には大きな差があります。それでも、まずはこのVision Frameworkだけで、食べ物の種類、つまり料理名をどこまで自動で入力できるかを試すことにしました。そのために作ったのが、次に説明するTier構造です。

## Tier構造の誕生：いちばん具体的な名前を拾うための階層

Visionが返すラベルには、具体度の差があります。同じ寿司の写真でも、具体的な `sushi` と、漠然とした `food` が同時に返ってきます。当然、より具体的な名前を優先して採用したくなります。そのためにラベルを段階分けしたのが、Tier構造のはじまりでした。

| Tier | 内容 | 例 |
|---|---|---|
| **Tier1** | 料理レベルの具体名 | `sushi`, `pizza`, `salmon`, `tempura` |
| **Tier2** | カテゴリ語や食材系 | `dessert`, `vegetable`, `lettuce`, `apple` |
| **Tier3** | 最も汎用的な食事ラベル | `food` |
| **Tier4** | 食器やカトラリー | `plate`, `bowl`, `chopsticks`, `cup` |

![Tier1からTier4までの階層と判定の流れを示した図](/images/jidoshokki-food-recognition/tier-structure.png)
*ラベルを具体度で4段に分け、上から順に拾っていく。*

ロジックは、Tier1にヒットすればその名前を使い、なければTier2、それもダメならTier3、という形でした。上から順に、より具体的な名前を拾っていきます。

いい名前を採用したい一心で、しきい値も上げました。中途半端な確信度で「パスタ」などと誤った名前を付けないために、Tier1のラベルは信頼度が0.7以上のときだけ料理名として採用する、という制約まで入れました。

このときはまだ、精度を詰めれば自動命名はいけると信じていました。

### 誤検知との戦い

ところがTierを使い込むほど、誤検知が次々に出てきます。誤検知を潰すたびに、ルールが増えていきました。ここからは、実際に出てきた誤検知と、それをどう塞いだのかを順番に紹介します。

一つ目は、水族館の魚の問題です。水族館の魚や動物園の鳥が、`fish` のようなTier1の食品ラベルにヒットして、食事として拾われてしまいます。そこで、Tier1単独では採用せず、カテゴリか食事か食器のいずれかが同時にヒットしたときだけ採用するようにしました。

![水族館で撮った魚の写真が食事候補として拾われてしまう例](/images/jidoshokki-food-recognition/misdetect-aquarium.png)
*水族館の魚が `fish` にヒットして、食事候補に紛れ込む。*

二つ目は、畑や食材だけの写真の問題です。もともとは、りんごやレタスのような果物と野菜のラベルもTier1に入れていました。ところが、畑や果樹園、あるいはまな板の上の食材だけを撮った写真で単独ヒットして、料理として拾われてしまいます。そこで、果物と野菜、ハーブといった食材系のラベルはTier1からTier2に格下げし、単独では料理名として採用しないようにしました。表のTier2に `lettuce` や `apple` が並んでいるのは、この格下げの結果です。

![畑の野菜や果樹園の果物が食事候補として拾われてしまう例](/images/jidoshokki-food-recognition/misdetect-produce.png)
*畑の野菜や果樹園の果物が、単独で食材ラベルにヒットしてしまう。*

三つ目は、空テーブルの問題です。もともと食器はTier3に混ざっていて、カップだけ、あるいは皿だけが写ったカフェの空テーブルが、食事と判定されてしまいました。そこで食器をTier4として分離し、精度を優先するモードでは、しきい値を超えた食器ラベルが2個以上あって、なおかつ料理か食材か食事のラベルが同時に写っていることを要求するようにしました。食卓のセッティングや食器棚だけの写真を、あとから塞いだかたちです。

![カフェの空テーブルが食事として判定されてしまう例](/images/jidoshokki-food-recognition/misdetect-empty-table.png)
*カップや皿だけの空テーブルが、食事として拾われてしまう。*

四つ目は、人物の早期除外です。家族や友人と撮った写真は、料理が写っていても人物が主役のことが多くあります。そこで、`adult` や `baby`、`bride` といった17種類の人物ラベルがしきい値以上だった場合は、料理が同時にヒットしていても食事とは認識せず、早い段階で処理を打ち切るようにしました。

```swift
if currentBlacklistPersonEnabled,
   let bestPerson = bestLabel(in: observations, predicate: isPersonLabel),
   bestPerson.confidence >= threshold {
    return ClassifyResult(recognized: nil, topLabel: bestPerson.identifier, ...)
}
```

![人物が主役の写真で、料理が写っていても除外される例](/images/jidoshokki-food-recognition/misdetect-person.png)
*人物が主役の写真は、料理が写っていても食事とは判定しない。*

これらの誤検知を一つずつ塞いでいくうちに、「食べ物かどうか」はかなり当たるようになりました。ところが、肝心の「正しい料理名を出す」のほうは、最後まで安定しませんでした。

## 最後の決断：名前を付けるのを諦めた

`udon` も `soba` も `onigiri` も `tonkatsu` も、Visionの辞書には存在しません。ラーメンもうどんも、結局は `food` か `pasta` に落ちてしまいます。Tier1のしきい値をどれだけ調整しても、日本食の名前そのものが当たらない以上、どうにもなりませんでした。

そして気づきます。誤った料理名を自動で入力することは、かえってユーザー体験を損なってしまいます。「ラーメン」の写真に「パスタ」と付くくらいなら、何も付いていないほうがましです。

そこで最後の最後に、方針を切り替えました。

> 名称ポリシー（諦めの結果たどり着いた背骨）
> スキャンではいっさい具体名を入れず、表示名は空の文字列、つまり「仮」表示にする。
> Visionは「これは食べ物か否か」の判定だけに使い、名前はユーザーが付ける。

この切り替えで、Tier構造の役割も大きく変わりました。もともとは「いい名前を取り出す」ための階層でしたが、名前を捨てたことで、「食べ物か否かを判定するためだけの階層」に生まれ変わりました。いま振り返ると、この4段はラベルの具体度そのものではなく、そのラベル1つをどれだけ信用してよいかという、証拠の強さの順に並んでいます。Tier1の具体名でも単独では信用せず、複数の段が同時にヒットしたときにだけ食事だと判断します。誤検知を一つずつ塞いだ結果として、この重み付けにたどり着きました。積み上げてきた誤検知対策、つまり水族館の魚や畑の食材、空テーブル、人物の除外は、判定器としてそのまま活きました。名前の自動入力という目標は捨てましたが、そこまでに作ったものが無駄になったわけではありません。

![スキャン結果が名前なしの「仮」表示になっている画面](/images/jidoshokki-food-recognition/scan-tentative-name.png)
*スキャン結果は具体名を入れず、オレンジの「仮」表示にする。名前はユーザーが付ける。*

### 名前を諦めた代わりに、手入力を限界まで楽にする

ユーザーが名前を付けると決めた以上、その手間をできるだけ小さくする必要があります。そこで命名を諦めたのと同じタイミングで、入力の補助を一気に強化しました。

一つは、かな入力のサジェストです。「や」と打つと「焼肉」を提案するように、ひらがなから漢字の料理名を絞り込めるようにしました。もう一つは、組み込み辞書の拡充です。定番の料理を約70件から約230件まで増やしました。さらに、履歴やお気に入りをワンタップで入力できるチップも用意しました。よく食べるものは、1タップで記録できます。

![かな入力のサジェストと、履歴やお気に入りのワンタップ候補チップ](/images/jidoshokki-food-recognition/name-input-assist.png)
*「や」と打つと「焼肉」が候補に出る。よく食べるものはチップから1タップで入る。*

AIが名前を当てることは諦めましたが、人が名前を付けるのは一瞬で済む、という状態には持っていけました。

## おわりに

AppleのVision Frameworkだけで写真から食べ物の種類まで判定できるか。試した結果は、「食べ物か否かは判定できるが、種類、つまり料理名までは安定して当てられない」でした。「ラーメンを検索できた」という体験からはじまり、肝心の検索機能は使えず、料理名の自動入力も最後には諦めました。振り返ると、諦めの連続でした。けれどそのたびに、できることの範囲に形が削られていって、結果として地に足のついたアプリになったように思います。

「重複した食事写真をどうまとめるか」「スキャンのパイプラインの全体像」あたりは、また別の記事で書こうと思います。
