## はじめに：きっかけは「ラーメン」検索

ある日、iPhoneの写真アプリの検索窓に何気なく「ラーメン」と打ってみたところ、過去に撮ったラーメンの写真がずらっと出てきました。自分では何のタグ付けもしていないのに、写真アプリが写真の中身を理解して食べ物を見つけてくれています。この挙動に、地味に感動しました。

当時の自分は、タンパク質と脂質、炭水化物を記録するPFC管理をやろうとしては毎回サボって続かない、ということを繰り返していました。多くの記録アプリは、食べる前に撮る、手で入力する、という流れを前提にしています。その手間が面倒で挫折してしまいます。

そこで思いついたのが、写真アプリの検索の仕組みを使えば、すでに撮ってあるアルバムの写真から食事を自動で拾って、後からまとめて記録できるのではないか、というアイデアです。これが「自動食記（じどうしょっき）」の出発点になりました。

ところが、作りはじめてすぐに最初の大きな誤算にぶつかります。

## 最初の誤算：あの検索APIは開発者に開放されていない

写真アプリで「ラーメン」を検索できるのは、端末の中で写真を解析して、写っているシーンや物体を認識しているからです。Appleはこのオンデバイスのシーン解析について[研究内容を公開](https://machinelearning.apple.com/research/on-device-scene-analysis)していて、写真を一枚ずつ端末内のニューラルネットワークで分類し、その結果を写真の検索などに使っていることがわかります。

ただ、写真アプリの検索を支えるこの高精度なモデルは、そのままの形では開発者に開放されていません。アプリの側から「ラーメンの写真をください」と頼むことはできません。地味に感動したあの体験を、自分のアプリでそのまま再現することはできませんでした。

代わりに使ったのが、iOS標準のVision Framework、具体的には [`VNClassifyImageRequest`](https://developer.apple.com/documentation/vision/vnclassifyimagerequest) です。これは汎用の画像分類器です。`knownClassifications(forRevision:)` で取得できるラベルを数えてみると、revision 2 では1303個ありました。完全にオンデバイスで動き、無料で、API制限もありません。写真をいっさい外部に送らないため、プライバシーの面でも安心して使えます。

条件だけを見ると申し分ないのですが、Appleの非公開モデルと比べると精度には大きな差があります。それでも最初は、諦めずに「料理名の自動入力」を実装しようとしていました。そのために作ったのが、次に説明するTier構造です。

## Tier構造の誕生：いちばん具体的な名前を拾うための階層

Visionが返すラベルには、具体度の差があります。同じ寿司の写真でも、具体的な `sushi` と、漠然とした `food` が同時に返ってきます。当然、より具体的な名前を優先して採用したくなります。そのためにラベルを段階分けしたのが、Tier構造のはじまりでした。

| Tier | 内容 | 例 |
|---|---|---|
| **Tier1** | 料理レベルの具体名 | `sushi`, `pizza`, `salmon`, `tempura` |
| **Tier2** | カテゴリ語や食材系 | `dessert`, `vegetable`, `lettuce`, `apple` |
| **Tier3** | 最も汎用的な食事ラベル | `food` |
| **Tier4** | 食器やカトラリー | `plate`, `bowl`, `chopsticks`, `cup` |

ロジックは、Tier1にヒットすればその名前を使い、なければTier2、それもダメならTier3、という形でした。上から順に、より具体的な名前を拾っていきます。

いい名前を採用したい一心で、しきい値も上げました。中途半端な確信度で「パスタ」などと誤った名前を付けないために、Tier1のラベルは信頼度が0.7以上のときだけ料理名として採用する、という制約まで入れました。

このときはまだ、精度を詰めれば自動命名はいけると信じていました。

### 誤検知との戦い

ところがTierを使い込むほど、誤検知が次々に出てきます。誤検知を潰すたびに、ルールが増えていきました。

一つ目は、水族館の魚の問題です。水族館の魚や動物園の鳥が、`fish` のようなTier1の食品ラベルにヒットして誤検知されてしまいます。そこで、Tier1単独では採用せず、カテゴリか食事か食器のいずれかが同時にヒットしたときだけ採用するようにしました。

二つ目は、空テーブルの問題です。もともと食器はTier3に混ざっていて、カップだけ、あるいは皿だけが写ったカフェの空テーブルが、食事と判定されてしまいました。そこで食器をTier4として分離し、精度を優先するモードでは、しきい値を超えた食器ラベルが2個以上あることを要求するようにしました。

三つ目は、人物の早期除外です。家族や友人と撮った写真は、料理が写っていても人物が主役のことが多くあります。そこで、`adult` や `baby`、`bride` といった17種類の人物ラベルがしきい値以上だった場合は、料理が同時にヒットしていても食事とは認識せず、早い段階で処理を打ち切るようにしました。

```swift
if currentBlacklistPersonEnabled,
   let bestPerson = bestLabel(in: observations, predicate: isPersonLabel),
   bestPerson.confidence >= threshold {
    return ClassifyResult(recognized: nil, topLabel: bestPerson.identifier, ...)
}
```

ここまで作り込んで、「食べ物かどうか」はかなり当たるようになりました。ところが、肝心の「正しい料理名を出す」のほうは、最後まで安定しませんでした。

## 最後の決断：名前を付けるのを諦めた

`udon` も `soba` も `onigiri` も `tonkatsu` も、Visionの辞書には存在しません。ラーメンもうどんも、結局は `food` か `pasta` に落ちてしまいます。Tier1のしきい値をどれだけ調整しても、日本食の名前そのものが当たらない以上、どうにもなりませんでした。

そして気づきます。誤った料理名を自動で入力することは、かえってユーザー体験を損なってしまいます。「ラーメン」の写真に「パスタ」と付くくらいなら、何も付いていないほうがましです。

そこで最後の最後に、方針を切り替えました。

> 名称ポリシー（諦めの結果たどり着いた背骨）
> スキャンではいっさい具体名を入れず、表示名は空の文字列、つまり「仮」表示にする。
> Visionは「これは食べ物か否か」の判定だけに使い、名前はユーザーが付ける。

この切り替えで、Tier構造の役割も大きく変わりました。もともとは「いい名前を取り出す」ための階層でしたが、名前を捨てたことで、「食べ物か否かを判定するためだけの階層」に生まれ変わりました。積み上げてきた誤検知対策、つまり水族館の魚や空テーブル、人物の除外は、判定器としてそのまま活きました。名前の自動入力という目標は捨てましたが、そこまでに作ったものが無駄になったわけではありません。

### 名前を諦めた代わりに、手入力を限界まで楽にする

ユーザーが名前を付けると決めた以上、その手間をできるだけ小さくする必要があります。そこで命名を諦めたのと同じタイミングで、入力の補助を一気に強化しました。

一つは、かな入力のサジェストです。「や」と打つと「焼肉」を提案するように、ひらがなから漢字の料理名を絞り込めるようにしました。もう一つは、組み込み辞書の拡充です。定番の料理を約70件から約230件まで増やしました。さらに、履歴やお気に入りをワンタップで入力できるチップも用意しました。よく食べるものは、1タップで記録できます。

AIが名前を当てることは諦めましたが、人が名前を付けるのは一瞬で済む、という状態には持っていけました。

## おわりに

「ラーメンを検索できた」という体験からはじまり、肝心のAPIは使えず、料理名の自動入力も最後には諦めました。振り返ると、諦めの連続でした。けれどそのたびに、できることの範囲に形が削られていって、結果として地に足のついたアプリになったように思います。

「重複した食事写真をどうまとめるか」「スキャンのパイプラインの全体像」あたりは、また別の記事で書こうと思います。
