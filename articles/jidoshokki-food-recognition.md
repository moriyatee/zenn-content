## はじめに：きっかけは「ラーメン」検索

ある日、iPhoneの写真アプリの検索窓に何気なく「ラーメン」と打ったら、過去に撮ったラーメンの写真がズラッと出てきました。これに地味に感動したんです。**自分は何のタグ付けもしていないのに、写真の中身を理解して食べ物を見つけてくれている。**

当時の自分は、PFC管理（タンパク質・脂質・炭水化物の記録）をやろうとしては毎回サボって続かない、を繰り返していました。記録アプリは「食べる前に撮る」「手で入力する」が前提で、それが面倒で挫折する。

そこで思いついたのが、「**この写真検索の仕組みを使えば、もう撮ってあるアルバムの写真から食事を自動で拾って、後からまとめて記録できるんじゃないか**」というアイデアでした。これが「自動食記（じどうしょっき）」の出発点です。

ところが、作り始めてすぐに**最初の大きな誤算**にぶつかります。

## 最初の誤算：あの検索APIは、開発者には開放されていない

写真アプリで「ラーメン」が検索できるのは、Appleが内部に持つ**非公開の高精度モデル**（おそらくCLIP系）のおかげです。そして調べてわかったのは、**この高精度モデルは開発者向けには一切開放されていない**ということ。アプリから「ラーメンの写真をください」とは頼めないのです。

あの感動の体験を、そのまま自分のアプリで再現することはできませんでした。

代替として使ったのが、iOS標準の **Vision Framework**（`VNClassifyImageRequest`）です。

- 汎用画像分類器（revision 2 で約1303ラベル）
- **完全オンデバイス・無料・API制限なし**
- 写真を一切外部に送らないのでプライバシー的にも安心

条件だけ見れば最高なのですが、Appleの非公開モデルとは精度が段違い。それでも最初は、**諦めずに「料理名の自動入力」を実装しようとしていました。** そのために作ったのが、次に説明する**Tier構造**です。

## Tier構造の誕生：「いちばん具体的な名前を拾う」ための階層

Visionが返すラベルには具体度の差があります。同じ寿司の写真でも `sushi`（具体的）と `food`（漠然）が同時に返ってくる。当然、**より具体的な名前を優先して採用したい**。そのために段階分けしたのが始まりでした。

| Tier | 内容 | 例 |
|---|---|---|
| **Tier1** | 料理レベルの具体名 | `sushi`, `pizza`, `salmon`, `tempura` |
| **Tier2** | カテゴリ語・食材系 | `dessert`, `vegetable`, `lettuce`, `apple` |
| **Tier3** | 最も汎用的な食事ラベル | `food` |
| **Tier4** | 食器・カトラリー | `plate`, `bowl`, `chopsticks`, `cup` |

ロジックは「Tier1にヒットすればその名前を使う。なければTier2、ダメならTier3…」と、**上から具体的な名前を拾っていく**形でした。

「いい名前を採用したい」一心で、しきい値も上げました。中途半端な確信度で「パスタ」などと誤った名前を付けないために、**Tier1ラベルは信頼度0.7以上のときだけ料理名として採用する**という制約を入れたほどです。

このときはまだ、「精度を詰めれば自動命名はいける」と信じていました。

### 誤検知との戦い

ところがTierを使い込むほど、誤検知が次々に出てきます。**潰すたびにルールが増えていきました。**

**① 水族館の魚問題（Tier1単独を禁止）**

水族館の魚や動物園の鳥が `fish` のようなTier1食品ラベルにヒットして誤検知される。
→ Tier1単独では採用せず、カテゴリ/食事/食器のいずれかが同時にヒットしたときだけ採用。

**② 空テーブル問題（Tier4分離）**

もともと食器はTier3に混ざっていて、「カップだけ」「皿だけ」のカフェの空テーブルが食事判定されてしまう。
→ 食器を**Tier4として分離**し、精度優先モードでは「閾値超えの食器ラベルが2個以上」を要求。

**③ 人物ブラックリスト（早期除外）**

家族や友人と撮った写真は、料理が写っていても人物が主役のことが多い。
→ `adult`, `baby`, `bride` など**17種の人物ラベル**が閾値以上なら、料理が同時にヒットしていても食事と認識しない（早期return）。

```swift
if currentBlacklistPersonEnabled,
   let bestPerson = bestLabel(in: observations, predicate: isPersonLabel),
   bestPerson.confidence >= threshold {
    return ClassifyResult(recognized: nil, topLabel: bestPerson.identifier, ...)
}
```

ここまで作り込んで、「**食べ物かどうか**」はかなり当たるようになりました。けれど、肝心の「**正しい料理名を出す**」は最後まで安定しなかったのです。

## 最後の決断：名前を付けるのを、ついに諦めた

`udon` も `soba` も `onigiri` も `tonkatsu` も、Visionの辞書には存在しません。ラーメンもうどんも、結局 `food` か `pasta` に落ちる。Tier1の閾値をどれだけ調整しても、**日本食の名前そのものが当たらない**以上、どうにもなりませんでした。

そして気づきます。**誤った料理名を自動入力するのは、ユーザー体験を最も損なう。**「ラーメン」の写真に「パスタ」と付くくらいなら、何も付いていないほうがいい。

そこで最後の最後に、方針を切り替えました。

> **名称ポリシー（断念の結果たどり着いた背骨）**
> スキャンでは一切具体名を入れず、表示名は空文字（=「仮」表示）にする。
> Visionは「**これは食べ物か否か**」の判定だけに使い、名前はユーザーが付ける。

この決断で、**Tier構造の役割も変わりました**。もともと「いい名前を取り出す」ための階層だったものが、名前を捨てたことで「**食べ物か否かを判定するためだけの階層**」に生まれ変わったのです。積み上げた誤検知対策（水族館・空テーブル・人物除外）は、判定器としてそのまま活きました。

### 名前を諦めた代わりに、手入力を限界まで楽にする

「ユーザーが名前を付ける」と決めた以上、その手間を最小化する必要があります。そこで命名を断念したのと**同じタイミング**で、入力補助を一気に強化しました。

- **かな入力サジェスト**：「や」と打つと「焼肉」を提案するように、ひらがなから漢字の料理名を絞り込み
- **組み込み辞書の拡充**：定番料理を **約70件 → 約230件** に増量
- **履歴・お気に入りのワンタップチップ**：よく食べるものは1タップで入力

「AIが名前を当てる」のは諦めても、「**人が名前を付けるのは一瞬で済む**」状態には持っていけました。

## おわりに

「ラーメンを検索できた」感動から始まり、肝心のAPIは使えず、料理名の自動入力も最後に諦める——**諦めの連続**でした。けれどそのたびに「できることの範囲で価値を最大化する」形に削られて、地に足のついたアプリになった気がします。

「重複した食事写真をどうまとめるか」「スキャンのパイプライン全体像」あたりは、また別の記事で書こうと思います。
